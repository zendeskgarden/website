## Anatomy

### Parts of a table

![Parts of a table](figma:patterns-tables-basic-formatting-anatomy.png)

1. Columns are vertical groupings of information that categorize and prioritize data
1. Header rows describe the data provided in a column or row
1. Rows are horizontal groups of data that fall under each column

## Formatting

### Size

Tables come in three sizes: small, medium, and large.

- Medium is the default size and accommodates information density in most cases
- Large is used when [Avatars](/components/avatar) are introduced into rows to create additional breathing room between
  row content
- Small can be used when more information density is needed

#### Layout

There are ways to design a table to make it easier to see and find information:

- A table must have at least 2 columns, otherwise use a [List](/components/lists) element
- Use horizontal lines to reduce visual noise in the table
- White space signals a different column
- You can specify max-width for columns but it's not required
- Don't place elements relating to the data outside of the table

#### Alignment

How text is displayed can make a column easier to view, use, and manage. Left align all text for greatest scannability,
with some exceptions for quantitative data.

![Cell Alignment](figma:patterns-tables-basic-formatting-cell-alignment.png)

| <b>Cellc content</b>                                                                                        | <b>Alignment</b> |
| ----------------------------------------------------------------------------------------------------------- | ---------------- |
| 1. Text                                                                                                     | Left align       |
| 2. Categorical data that uses numbers (for example, [dates](/content/grammar#date-and-time), phone numbers) | Left align       |
| 3. Quantitative data                                                                                        | Right align      |
| 4. [Tags](/components/tags) or [Avatars](/components/avatar)                                                | Left align       |
| 5. Icons                                                                                                    | Center align     |

## Avatars, icons, and tags

### Avatars

[Avatars](/components/avatar) help represent a person or brand in tables. They can display icons or images.

### Size

Avatar sizes for tables include default and small.

![Small avatars in a Default table](figma:patterns-tables-basic-formatting-small-avatars-default-table.png)

- Use a small Avatar in a default table when more density is needed when displaying information
  about many users at once
- Default size tables with an Avatar have a height of 52px with 10px of top and bottom padding and 12px of
  left and right padding

![Default avatars in a Large table](figma:patterns-tables-basic-formatting-default-avatars-large-table.png)

- Use a default Avatar in a large size Table in contexts where the user data is the main object of the table,
  and if there are multiple items of metadata associated with the user
- Large size tables with an Avatar have a height of 60px with 10px of top and
  bottom padding and 12px of left and right padding

### Icons

There are 2 types of icons that are used in tables: informational icons and icon buttons.

![Table icons](figma:patterns-tables-basic-formatting-action-icons.png)

1. Informational icons are icons that help communicate the state of the cell value. Informational icons must include
   a `title` attribute and `aria-label` to provide mouse and keyboard users with more context about the icon’s meaning.
1. [Icon Buttons](/components/icon-button) allow the user to complete an action, for example opening an overflow menu

### Tags

[Tags](/components/tags) let users categorize cell content using a keyword.

![Tags in table](figma:patterns-tables-basic-formatting-tags-anatomy.png)

1. The default size for Tags in table cells is small. Tags are left aligned with 4px of spacing between each tag.
1. If there are more Tags than can be displayed in a cell, use an Anchor to show the number of hidden Tags.
   See [Wrapping and Truncation](/patterns/tables/basic-formatting#wrapping-and-truncation) for more information.
1. If a Tag exceeds the width of a cell, the Tag can be truncated.
   See [Wrapping and Truncation](/patterns/tables/basic-formatting#wrapping-and-truncation) for more information.
1. Tags can be inline with other data about a user. The Tag should be inline with the data it is related to with
   the same 4px spacing between the data and Tag.

## Wrapping and truncation

### Text

Text wrapping is the default and recommended behavior. Ensure the cells in that row are top aligned to help with
scannability. Keep in mind, truncation in a table can make it harder for users to scan.
Use an [Ellipsis](/components/typography#ellipsis-text) to truncate long text at the end of the first line and provide
a title attribute that appears on hover.

![Wrapping and truncation](figma:patterns-tables-basic-formatting-wrap-and-truncation.png)

1. Header and data cells should be concise
1. Text wrapping is the default and recommended behavior. Ensure the cells in the row are top aligned to help with
   scannability.
1. Keep in mind, truncation in a table can make it harder for users to scan. Use an
   [Ellipsis](/components/typography#ellipsis-text) to truncate long text at the end of the first line and
   provide a title attribute that appears on hover.

#### Header cells

Headers categorize and organize the information within the table.

- Ensure header cells accurately describe the data contained in the rows and columns
- Use one or two nouns that quickly help people understand what data is in the table
- Avoid using verbs to prevent confusion with CTAs
- Header cell labels shouldn't create sentences with data cell content
- Use sentence case
- Header cells are marked up with `<th>` to make them accessible
- Ensure [visually-hidden header cell content](/components/table#draggable) is accessible for screen readers

Internationalization could result in text expansion. If the text length in a header cell exceeds the specified width of
a column, there are a few options:

- Truncate with an [Ellipsis](/components/typography#ellipsis)
- Wrap the content. In the case of headers, content should be bottom aligned.
- Ignore the specified width and stretch the column width to accommodate the text. This could trigger horizontal
  scrolling for the table.

#### Data cells

Data cells contain the actual data and are located below header cells.

- Ensure the content is detailed yet concise
- Avoid repeating the words used in the header to help with scannability
- Use sentence case
- Data cells are marked up with `<td>` to make them accessible

### Tags

#### Truncation

Any tags that exceed the width of the cell, use an [Ellipsis](/components/typography#ellipsis)
to truncate long text and provide a title attribute that appears on hover.

![Tags truncation](figma:patterns-tables-basic-formatting-tags-truncation.png)

#### Overflow

If there are greater than two tags, use a [Button](/components/button) with the `isLink` prop indicating the
number of additional tags.

![Tags overflow](figma:patterns-tables-basic-formatting-tags-overflow.png)

- When the user interacts with the button, it will open a [Tooltip Modal](/components/tooltip-modal) with `auto`
  placement set, ensuring it remains viewable within the viewport
- When toggling the tag list, ensure the appropriate
  [ARIA attributes](https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/Attributes/aria-expanded) are included
- For localization, create two strings for the button text: one for the "+ {number}" and another for the "more" string
- If the number of tags becomes unwieldy within the [Tooltip Modal](/components/tooltip-modal), include an action that
  allows the user to view all tag information in a separate view

![Tags overflow expanded](figma:patterns-tables-basic-formatting-tags-overflow-expanded.png)

#### Grouped rows

When creating additional categorization of similar items, use [grouped rows](/components/table#grouped-rows).

- They should be used only when the groups will contain 10 or more items
- Avoid overuse to prevent busy layouts that are harder to understand
- Don’t use striped and grouped rows at the same time

![Grouping](figma:patterns-tables-basic-formatting-grouping.png)

### Row actions

There are multiple locations within a row to perform actions. Common actions include View, Edit, Delete
or a combination of these within a single row.

- If linking to another page or a new tab, use an [Anchor](/components/anchor)
- If opening a Drawer/Sheet/Modal in the same view, use a [Button](/components/button) with the `isLink` prop

#### View and Edit

"View" or "Edit" actions are to be contained within the data cells of a row in addition to the overflow menu.

![View actions](figma:patterns-tables-basic-formatting-view-actions.png)

1. The first "View" or "Edit" action is contained within the first column and will take the user to additional
   view or edit detail on that given item
1. The overflow menu includes an additional "View" or "Edit" action, providing a second way to access the same
   content for viewing or editing

#### Additional actions

All additional actions are to be placed inside the [overflow menu](/components/table#overflow).

- Group complementary actions inside the menu, using separators to create visual hierarchy
- For destructive actions, place a separator line above and use the danger styling

![Overflow actions](figma:patterns-tables-basic-formatting-overflow-actions.png)

#### Screen readers

Ensure elements for the rows are clearly labeled and accessible.

![Overflow tooltip](figma:patterns-tables-basic-formatting-overflow-tooltip.png)

#### Pagination

Pagination separates table rows into pages and allows users to navigate between those pages. Our table component is
paired with the [Pagination](/components/pagination) component to create a consistent experience.

- Cursor pagination is the default and recommended approach [due to performance](https://support.zendesk.com/hc/en-us/articles/4408846180634)
- Cap a single page view at 100 rows

##### Cursor pagination

Use [Cursor pagination](/components/pagination#cursor-pagination) for real-time data.

- This includes when new records are added frequently or in data sets where the latest results are first
- There is no concept of the total number of pages in the set
- Use “Previous” and “Next” as pagination labels

![Pagination Cursor](figma:patterns-tables-basic-formatting-pagination-cursor.png)

##### Offset pagination

Use [Offset pagination](/components/pagination#offset-pagination) when you know the number of records or where you are
in the data set.

- Ensure the margin between the table component and the pagination is 20px

![Pagination Offset](figma:patterns-tables-basic-formatting-pagination-offset.png)

##### Select all for pagination

In paginated tables, you can select all checkboxes but that selection only affects the current page being shown.

- Users should only be able to take action on what they can see

![Pagination Selecting All](figma:patterns-tables-basic-formatting-pagination-selecting-all.png)

#### Infinite scrolling

In cases where the database supports infinite scrolling, there is no need to say how many pages of data there are.
If you have an infinitely scrolling table, you should also offer [sorting](/patterns/tables/basic-formatting#sorting) to
help users navigate the data.

##### Select all for infinite scrolling

In infinitely scrolling tables, you can only manually select items.

##### Vertical scrolling

For tables with more rows than can be displayed in the viewport, the header row is fixed at the top of the viewport.
Keeping the column headers within view while scrolling helps users maintain context.

#### Sorting

Sorting is implemented by selecting arrows in table headers.

- Clicking the header text will sort by ascending order (A→Z, 1→9, Date and Time, Priority (High→Low))
- Clicking again will reverse the order
- Clicking a third time will remove sorting

If any ordering is applied to a table including default ordering, the sorting icon should be present. This indicates how
the table is being ordered and also creates a clear affordance to control.

### Loading

Skeleton screens are used to indicate that a screen is loading. They are perceived as being shorter in duration when
compared against a blank screen or a spinner.

- Implement the [Skeleton](/components/skeleton) component in each cell up to three rows
- For a table with three or more rows, show the first three rows as skeletons with the header cells being shown at all
  times to provide instant context of incoming tabular data
- The width of the Skeleton should match the width of the intended cell content, while the Skeleton’s height will be 60%
  of the font size

## Accessibility

### General considerations

All Garden [Tables](/components/table) are built with accessible markup.

### Screen reader order

People using screen readers can have the row and column cells read aloud as they navigate through the table. Screen
readers speak one cell at a time and reference the associated header cells, so the reader doesn’t lose context.
